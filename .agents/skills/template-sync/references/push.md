# template-sync: Push 型（派生 → テンプレート）

[SKILL.md](../SKILL.md) · パターン・原則は [shared.md](shared.md)。

派生で効いた変更を、`yellow-seed/template` に**GitHub Issue で提案**する。派生側からテンプレへ直接 PR を開かなくてよい（リポジトリ跨ぎの作業負荷を避ける）。

実装・マージはテンプレのメンテナが Issue を見て `yellow-seed/template` 上で行う。マージ後、各派生は通常の **Pull**（[pull.md](pull.md)）で追随する。

## 還流してよい候補の条件

- **汎用化できる**（プロジェクト名・秘密・内部 URL が残らない）
- **ライセンス・社内規程**で公開テンプレに載せて問題ない
- 可能なら**小さな単位**で説明しやすい（Issue を分割してもよい）

## Issue に書く内容（テンプレート）

作成先: **`yellow-seed/template`**（例: Web の New Issue、`gh issue create --repo yellow-seed/template`）

1. **要約タイトル**（変更の種類がわかる一行）
2. **背景・目的**（派生でどんな問題を解いたか）
3. **出典**
   - 派生リポジトリ URL
   - コミット SHA またはブランチ名
   - 対象パス（テンプレと揃えると取り込みが速い）
4. **提案内容**
   - テンプレに取り込みたい差分の説明（箇条書きでよい）
   - **差分そのもの**: フェンス付き unified diff、`git format-patch` の貼り付け、gist へのリンクなど、運用で決めた形で添付
   - 取り込み時の注意（シンボリックリンクや `.agents` 配置など）
5. **受け入れの目安**（任意: 影響範囲、Pull 派生への周知要否）

機微情報は Issue に貼らない。必要なら抽象化した例だけにする。

## 手順の流れ（派生側）

1. 還流対象をパス／コミット単位に特定し、[shared.md](shared.md) の「同期しない」に該当しないか確認する
2. 派生固有情報を除き、説明文と diff を準備する
3. **`yellow-seed/template` に Issue を作成**する（テンプレートに合わせたラベルがあれば付与）
4. テンプレ側で対応がマージされるまでフォロー（質問への回答、diff の修正など）
5. マージ後、必要なら派生へ「テンプレ追随」のメモやタスクを残す

## 載せないことが多いもの

アプリ固有の依存バージョン、環境変数の値、製品ドメインに結びついた README、組織だけの運用。

## メンテナ向けメモ（参考）

Issue に貼られた内容を、`yellow-seed/template` のローカル clone / worktree で cherry-pick・手動適用するのはメンテナ側の作業。本スキルの Push パターンは **Issue で依頼するまで**を既定とする。

## Push チェックリスト

- [ ] 秘密・固有名・環境依存を Issue に載せていない／抽象化した
- [ ] 出典（リポジトリ・SHA・パス）を書いた
- [ ] 提案内容・差分がレビュー可能な単位になっている
- [ ] `yellow-seed/template` 上に Issue を作成した
- [ ] マージ後、派生の Pull 追随が必要なら追跡できるようにした
